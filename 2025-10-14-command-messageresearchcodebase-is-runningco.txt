  4. Sessions vs Tokens vs Connections

  The Key Distinction

  Connection (Client) = Process-level connection to MCP server
  - Lives in SessionManager.sessions sync.Map
  - Keyed by: deploymentID + clientScope
  - Represents a nmcp.Client connected to the actual MCP server process/container
  - Created once per deployment + scope combination
  - Reused across multiple HTTP requests from same session

  Session = User's conversation state with MCP server
  - Stored as v1.MCPSession Kubernetes resource
  - Keyed by: session ID (from Mcp-Session-Id header)
  - Contains serialized nmcp.SessionState (initialization details, capabilities)
  - Persists across connection drops
  - Updated on every message exchange

  Token = Authentication credential
  - First-level tokens: JWT access token + refresh token for Client‚ÜíObot auth
    - Stored in OAuthToken Kubernetes resource
    - 10 minute expiration, refreshable
  - Second-level tokens: OAuth token for Obot‚ÜíBackend auth
    - Stored in MCPOAuthToken database table
    - Encrypted at rest
    - Auto-refreshed by oauth2 library

  6. Special Considerations for VMCP Design

  Based on this architecture, here are key insights for implementing Virtual MCP (composite)
  servers:

  Deployment Strategy

  Option 1: Composite Backend
  - Create a new backend type that wraps multiple MCP clients
  - SessionManager would create one "virtual deployment" that internally manages N real
  deployments
  - Pro: Fits existing architecture cleanly
  - Con: Complex credential/OAuth management across components

  Option 2: Proxy Layer
  - Create a containerized MCP server that acts as a proxy/aggregator
  - Deploy like any other containerized runtime
  - Internally connects to component MCP servers
  - Pro: Reuses existing deployment/connection logic
  - Con: Requires new proxy implementation

  Tool/Resource Name Transforms

  The gateway currently uses the raw tool names from MCP servers. For VMCP:
  - Need bidirectional name mapping: {vmcp_tool_name} ‚ÜĒ {component_server}:{real_tool_name}
  - Apply in OnMessage before calling Exchange() (client‚Üíserver)
  - Apply in result before calling Reply() (server‚Üíclient)
  - Store mappings in VMCP catalog entry manifest

  Multi-Level OAuth

  When VMCP has multiple component servers requiring OAuth:
  - Need to track OAuth state per component: (userID, vmcpID, componentMCPID)
  - CheckForMCPAuth would need to check all components
  - Return combined OAuth URL or redirect through each provider sequentially
  - TokenStore would need hierarchical structure

  Session Management

  Sessions are tied to single MCP server currently. For VMCP:
  - Session state would need to track state per component server
  - Could maintain N sub-sessions (one per component)
  - Or create stateless VMCP proxy that doesn't need session storage

  Audit Logging

  Current design logs per-server. For VMCP:
  - Top-level audit log for VMCP call
  - Optional sub-audit logs for component server calls (hidden from user)
  - Need to track which component server handled which tool call

  - Contains: User authentication state, session cookie value
  - Lifetime: From login until logout
  - Used for: First-level OAuth authentication

  User ‚Üí Logs into Obot UI
       ‚Üź Sets cookie: obot_access_token (contains session ID)
       ‚Üí Redirected to: /oauth/callback/{oauth_auth_request_id}/{mcp_id}

  Identifiers created:
  1. Obot Session ID: From the login cookie (oauth2-proxy ticket)
  2. Hashed Session ID: SHA-256 hash of session ID
  3. Authorization Code: {random_text}{random_text} - two concatenated random strings
  4. Hashed Authorization Code: SHA-256 hash of the code

  Storage updates:

  SessionCookie (gateway database):
  - Primary Key: (HashedSessionID, AuthProviderNamespace, AuthProviderName)
  - Stores: Encrypted cookie value, UserID

  v1.OAuthAuthRequest (Kubernetes):
  - Updates with: HashedAuthCode, UserID, HashedSessionID, auth provider info

  Authorization code returned to client:
  302 Redirect to: {client_redirect_uri}?code=AUTHORIZATION_CODE&state=CLIENT_STATE

  ---
  Phase 3b: Second-Level OAuth (If MCP Server Requires It)

  When does this happen? If mcp_id is provided AND the MCP server is a remote server requiring
  OAuth (e.g., connecting to Google Drive MCP server that needs Google OAuth).

  What happens:
  Obot ‚Üí Tries to connect to MCP server
       ‚Üź MCP server returns: 401 with WWW-Authenticate header (OAuth required)
       ‚Üí Generates state and redirects user to: Third-party OAuth (e.g., Google)

  Identifiers created:
  1. MCP OAuth State: {random_text} - for correlating the callback
  2. Hashed State: SHA-256 hash for database lookup
  3. PKCE Verifier: Random string for PKCE flow

  Storage: MCPOAuthToken in gateway database
  - Primary Key: (MCPID, UserID)
  - Stores (encrypted): State, Verifier, ClientID, ClientSecret, OAuth endpoint URLs
  - Links back: OAuthAuthRequestID (points to the first-level auth request)
  - Tokens NOT stored yet - just the OAuth config and state

  Also cached in-memory: stateCache maps state ‚Üí channel for async callback

  User redirected to third-party:
  302 Redirect to: https://accounts.google.com/oauth/authorize?state=MCP_STATE&...

  ---
  Phase 3c: Second-Level Callback

  What happens:
  User ‚Üí Authenticates with Google (or other provider)
       ‚Üź 302 Redirect to: /oauth/mcp/callback?state=MCP_STATE&code=THIRD_PARTY_CODE

  Processing:
  1. Look up state in cache or database (by hashed state)
  2. Exchange THIRD_PARTY_CODE for access/refresh tokens with Google
  3. Store encrypted tokens in MCPOAuthToken database record
  4. Retrieve the OAuthAuthRequestID from the token record
  5. Load the first-level v1.OAuthAuthRequest resource
  6. Generate NEW authorization code for first-level flow
  7. Update OAuthAuthRequest with new hashed code

  Why new auth code? The first code was already hashed and can't be recovered!

  User redirected back to MCP client:
  302 Redirect to: {client_redirect_uri}?code=NEW_AUTHORIZATION_CODE&state=CLIENT_STATE

  ---
  Phase 4: Token Exchange

  What happens:
  MCP Client ‚Üí POST /oauth/token
              grant_type=authorization_code
              code=AUTHORIZATION_CODE
              code_verifier=... (if PKCE used)
            ‚Üź Returns: access_token (JWT), refresh_token

  Processing:
  1. Hash the submitted code
  2. Look up v1.OAuthAuthRequest by HashedAuthCode field
  3. Delete the OAuthAuthRequest (one-time use!)
  4. Verify PKCE if code_challenge was used
  5. Verify session cookie still exists (user hasn't logged out)
  6. Generate JWT access token (10-minute expiry)
  7. Generate refresh token
  8. Create v1.OAuthToken with SHA-256 hash of refresh token as name

  Identifiers created:
  1. Access Token: Signed JWT containing user info, expires in 10 min
  2. Refresh Token: Random text
  3. Hashed Refresh Token: SHA-256 hash used as K8s resource name

  Storage: v1.OAuthToken in Kubernetes
  - Name: Hash of refresh token
  - Stores: ClientID, UserID, HashedSessionID, resource URL
  - Does NOT store actual tokens - only metadata

  ---
  Phase 5: Using Tokens for MCP Requests

  What happens:
  MCP Client ‚Üí GET /api/mcp-gateway/{mcp_id}
              Authorization: Bearer {ACCESS_TOKEN}
              Mcp-Session-Id: {mcp_protocol_session_id}

  First-Level Token Validation:
  - JWT validated using JWKS public key
  - Claims extracted: UserID, session, auth provider

  Second-Level Token Usage (if remote MCP with OAuth):
  - Gateway calls tokenStore.ForUserAndMCP(userID, mcpID)
  - Loads MCPOAuthToken from database
  - Decrypts access/refresh tokens
  - Passes to MCP client library
  - Library automatically refreshes if expired

  MCP Protocol Session:
  - Separate from OAuth - tracks MCP state
  - Stored in v1.MCPSession Kubernetes resource
  - Updated on every MCP message

  ---
  How Identifiers Relate to Each Other

  Client Registration:
    client_id ‚Üí v1.OAuthClient.Name

  First-Level OAuth:
    client_id ‚Üí OAuthAuthRequest.Spec.ClientID
    client_state ‚Üí OAuthAuthRequest.Spec.State (passed through)
    
  User Authentication:
    session_cookie ‚Üí SessionCookie.Cookie (encrypted)
    SHA256(session_id) ‚Üí OAuthAuthRequest.Spec.HashedSessionID
                       ‚Üí OAuthToken.Spec.HashedSessionID
    
  Authorization Code:
    SHA256(code) ‚Üí OAuthAuthRequest.Spec.HashedAuthCode (index for lookup)
    
  Second-Level OAuth:
    mcp_state ‚Üí MCPOAuthToken.State (encrypted)
    SHA256(mcp_state) ‚Üí MCPOAuthToken.HashedState (unique index)
    oauth_auth_request_id ‚Üí MCPOAuthToken.OAuthAuthRequestID (links back)

  Third-Party Tokens:
    access_token ‚Üí MCPOAuthToken.AccessToken (encrypted)
    refresh_token ‚Üí MCPOAuthToken.RefreshToken (encrypted)

  First-Level Tokens:
    access_token ‚Üí JWT (signed, not stored)
    SHA256(refresh_token) ‚Üí OAuthToken.Name (resource name)

  ---
  Storage Summary

  | What                   | Where                     | How              |
  |------------------------|---------------------------|------------------|
  | Client credentials     | v1.OAuthClient (K8s)      | bcrypt hash      |
  | Authorization request  | v1.OAuthAuthRequest (K8s) | Hashed codes     |
  | Session cookie         | SessionCookie (DB)        | Encrypted        |
  | Refresh token metadata | v1.OAuthToken (K8s)       | Hashed name      |
  | JWT access token       | Not stored                | Signed JWT       |
  | MCP OAuth tokens       | MCPOAuthToken (DB)        | Encrypted        |
  | MCP protocol session   | v1.MCPSession (K8s)       | Serialized state |

  ---
  Key Takeaways

  1. Two session types: MCP protocol session (header) vs Obot user session (cookie) -
  completely independent!
  2. OAuthAuthRequest is the central coordinator:
    - Created at authorization start
    - Updated after user auth
    - Links to session via HashedSessionID
    - Deleted after one-time code exchange
  3. Hashing everywhere:
    - Auth codes: Never stored in plaintext
    - Refresh tokens: Hash used as K8s resource name
    - Session IDs: Hashed for privacy
    - States: Hashed for database lookup
  4. Two-level linking: MCPOAuthToken.OAuthAuthRequestID connects second-level MCP OAuth back
  to first-level client OAuth
  5. Token refresh rotation: Both levels rotate tokens on refresh - old deleted, new created
  6. Encryption only for second-level: First-level uses hashing + JWTs, second-level uses AES
  encryption for third-party tokens
